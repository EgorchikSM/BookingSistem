# Практическое занятие №4: Реализация устойчивой веб-службы

## Описание проекта

Данный проект представляет собой учебную веб-службу на базе **ASP.NET Core**, моделирующую процесс бронирования переговорной комнаты. Основная цель работы — реализация системы, устойчивой к повторной доставке событий (идемпотентность) и частичным сбоям (компенсация), с развитыми механизмами наблюдаемости.

---

## 1. Машина состояний

Процесс бронирования включает в себя следующие состояния и переходы:

| Исходное состояние | Событие | Новое состояние | Описание |
| --- | --- | --- | --- |
| **New** | ПринятьЗаявку | **RequestAccepted** | Первичная регистрация намерения |
| **RequestAccepted** | Забронировать | **ResourceReserved** | Бронирование комнаты в календаре |
| **ResourceReserved** | ВыдатьДоступ | **AccessGranted** | Генерация ключа доступа (точка риска) |
| **AccessGranted** | Завершить | **Completed** | Успешное окончание процесса |
| **ResourceReserved** | Сбой доступа | **CompensationExecuted** | Откат (освобождение брони) |

---

## 2. Ключевые механизмы

### Идемпотентность

Для предотвращения побочных эффектов при повторной доставке событий используется **Ключ идемпотентности** (`IdempotencyKey`).

* **Реализация:** Служба хранит список обработанных ключей для каждого процесса.
* **Логика:** При получении события проверяется, был ли данный ключ обработан ранее. Если да, состояние не меняется, возвращается текущий статус, а в метриках фиксируется повтор.

### Наблюдаемость и Диагностика

* **Сквозной идентификатор (Correlation ID):** Передается в заголовках запроса (`X-Correlation-ID`). Используется для объединения всех записей журнала (логгирование в Scope) и ответов API.
* **Логгирование:** Использована библиотека **Serilog**. Журналируются переходы состояний, срабатывание идемпотентности и запуск компенсаций.
* **Метрики:** Реализована служба `MetricsService`, собирающая счетчики:
* Успешные переходы (`success`)
* Ошибки (`error`)
* Повторные доставки (`idempotency`)
* Компенсации (`compensation`)



### Компенсация и Сбои

В системе реализован сценарий "Авария на шаге ВыдатьДоступ". Если внешняя система выдачи ключей возвращает ошибку, транзакция откатывается: состояние переходит в `CompensationExecuted`, что сигнализирует об отмене ранее забронированных ресурсов.

## 3. Проверки здоровья (Health Checks)

Реализованы два эндпоинта для мониторинга:

1. **Liveness (`/health/live`):** Проверка "живости" приложения.
2. **Readiness (`/health/ready`):** Проверка готовности к работе.
* *Критическая деградация:* Если количество компенсаций превышает 5, проверка готовности возвращает `Unhealthy`, сигнализируя о проблемах с внешними зависимостями.

## 4. Инструкция по запуску и тестированию

### Запуск

1. Откройте решение в Visual Studio.
2. Запустите проект (F5). Откроется страница **Swagger**.

### Тестирование сценариев в Swagger

#### Сценарий "Успех"

Отправьте последовательно 4 POST-запроса на `/api/Booking/event` с разными типами событий (`ПринятьЗаявку` -> `Забронировать` -> `ВыдатьДоступ` -> `Завершить`) и уникальными ключами идемпотентности.

#### Сценарий "Идемпотентность"

Отправьте один и тот же запрос дважды. Убедитесь в логах (консоль или файл), что зафиксирован повтор, а состояние не изменилось.

#### Сценарий "Компенсация"

Отправьте запрос на событие `ВыдатьДоступ`, добавив в заголовок `X-Correlation-ID` значение `fail`. Система имитирует сбой и перейдет в статус `CompensationExecuted`.

### Проверка показателей

Вызовите `GET /api/Booking/metrics`, чтобы увидеть актуальную статистику работы службы.

## 5. Использованные паттерны

* **State Machine:** Управление переходами.
* **Idempotent Receiver:** Защита от дублей.
* **Saga (Compensating Transaction):** Обработка частичных сбоев.
* **Dependency Injection:** Регистрация служб как "плагинов".
* **Health Check API:** Мониторинг состояния службы.

---

> **Вывод:** Реализованная служба полностью соответствует функциональному минимуму, обеспечивая прозрачность (через логи и метрики) и надежность (через идемпотентность и компенсацию) процесса бронирования.
